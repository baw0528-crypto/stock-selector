# stock-selector 設計レビュー依頼

**目的**: このドキュメントはFable 5への設計精査を依頼するための叩き台です。
実装の細部説明ではなく「なぜこう決めたか」「弱いと思っている点」
「まだ決めていない点」を中心に書いています。忌憚なく指摘してください。

**前提として明確にしておきたいこと**: このシステムは銘柄の一次スクリーニングと
レポート生成のみを行います。発注・自動売買は行いません。売買判断と執行は
すべて人間(利用者本人)が行います。設計レビューではこの役割分担を崩さない
範囲での改善案を求めています。

**対象市場について**: 当初は日本株・米国株の両対応を検討していましたが、
(1)米国株の方がセクターETF・データソースが整備されておりセクターローテーション
戦略と相性が良いこと、(2)個人開発として2市場を並行保守するコストが高いこと、
の2点から**米国株に対象を絞る**判断をしました。日本株関連コード
(`jquants_client.py`等)は将来の再拡張に備えて残していますが、現状は
デフォルトで無効化しています。この判断自体への異論があれば指摘してほしいです。

## 1. ゴールと非ゴール

**ゴール**
- 米国株を対象に、ファンダ・テクニカル・ニュース・セクターローテーションの
  4軸で一次候補を絞り込む
- 「今勢いのあるセクター」を判定し、その中の銘柄選定につなげる
  (セクターだけ見て終わりの既存アプリとの差別化ポイント)
- 絞り込んだ候補についてAIが「なぜ上位に来たか」を人間可読な形で説明する
- 選定基準(重み付け)を利用者が自分の投資スタイルに合わせて調整できる

**非ゴール**
- 発注・約定・ポジション管理は対象外
- 高頻度取引やミリ秒単位のレイテンシは想定しない(日次〜週次のバッチ運用)
- バックテストによる戦略の検証は現時点ではスコープ外(将来検討)

**Fableへの問い**: ゴール設定として過不足はないか。「一次スクリーニング」という
役割は、後段で人間が判断するという前提を考えると妥当な範囲か、それとも
中途半端(判断材料が足りないか、逆に人間の判断を過度に誘導しすぎるか)か。

## 2. 現状のアーキテクチャ

```
[sector ETF(SPDR等)]--+
                       |--> sector_rank --+
[yfinance]-------------+                  |
                                           v
                        data layer --> analysis layer --> scorer --> agent(Fable5) --> report.md
                                           ^
[Google News RSS]-------------------------+
```

- **sector layer**: セクターETFの相対強度(ベンチマーク比騰落率)を計算し、
  勢いのあるセクターを判定する(`sector_rank.py`)。`--sector-first`実行時のみ
  経由し、通常実行時はスキップされる
- **data layer**: データソースごとにクライアントを分離
  (`us_market_client.py`, `news_client.py`。`jquants_client.py`は現状未使用)
- **analysis layer**: ファンダ(`fundamentals.py`)とテクニカル(`technicals.py`)を
  それぞれ独立に0-100スコア化
- **scorer**: 3軸(ファンダ/テクニカル/ニュース)を重み付け線形合成して
  `total_score`を出す(`CandidateScore.compute_total`)。セクター強度自体は
  現状スコアに含めておらず、あくまで銘柄の絞り込み(フィルタ)にのみ使っている
- **agent**: 上位候補の定量データ+ニュース見出しをFable 5に渡し、
  定性的なサマリー・注目点・リスクを生成させる(`fable_synthesis.py`)
- **実行方式**: Claude Code上のスキル(`SKILL.md`)としてCLI(`screen.py`)を
  自然言語からラップして呼ぶ

**設計上の決定とその理由**
| 決定 | 理由 |
|---|---|
| データ取得と分析を層で分離 | データソースが増えても分析ロジックを変えずに済むようにするため |
| スコアは0-100に正規化して合成 | 単位の異なる指標(PER、ROE%、出来高比率等)を横並びで比較・重み付けできるようにするため |
| ニュースセンチメントはキーワードベースの簡易判定 | 外部センチメントAPIへの依存とコストを避け、最終判断はFable側の定性評価に委ねる設計にしたため |
| 発注機能を持たせない | 自動売買のリスク(暴走・誤発注)を構造的に排除するため。moomoo API Skillの「人間の最終確認必須」という設計思想を踏襲 |
| セクター強度をスコアに混ぜずフィルタとして使う | セクターの勢いと個別銘柄の質は別の情報だと考え、混ぜて1つの数値にすると
「セクターは弱いが銘柄は良い」ようなケースを機械的に排除してしまうため |
| 米国株に対象を絞る | データソースの充実度とセクターローテーション戦略との相性、保守コストの3点から判断 |

**Fableへの問い**:
- 層分離の粒度は適切か。過剰設計(YAGNI違反)になっていないか、
  逆に将来の拡張(証券会社追加など)に対して不十分な抽象化になっていないか
- 0-100正規化という設計は、指標間の非線形な関係(例:PERは10→20と
  90→100で意味が全く違う)を潰してしまっていないか
- 「発注機能を持たない」という制約は妥当か。もし将来発注を追加するとしたら、
  今のアーキテクチャのどこに境界線を引くべきか
- セクター強度を「フィルタ」に留めている設計は妥当か。それとも
  スコアの一部として重み付けすべきか

## 3. スコアリングロジックの詳細と妥当性への疑問

### 3.1 ファンダメンタルズスコア (`fundamentals.py`)

現在の実装:
- ROE: `30 + roe*3` を0-95にクリップ (ROE 10%で60点、20%で90点)
- 売上成長率: `50 + growth*2` を0-95にクリップ
- PER: `100 - (per-10)*2.5` を10-90にクリップ (低PERほど高得点)
- 利益率: `40 + margin*1.5` を0-90にクリップ
- 取得できた指標の単純平均

**疑わしいと思っている点**:
- 係数(×3, ×2, ×2.5, ×1.5)は完全に経験則・仮置きで、バックテストによる
  検証をしていない
- PERは業種によって適正水準が全く違う(グロース株とバリュー株を同じ基準で
  評価している)のに、業種補正が一切ない
- 指標が1つしか取れない銘柄(例: ROEのみ)と4つ揃っている銘柄が同じ
  「平均」ロジックで扱われ、情報量の差が評価に反映されない

**Fableへの問い**: この係数設計の妥当性、業種補正の要否、
「データが少ない銘柄ほど信頼区間が広いはず」という点をスコアに
どう反映すべきか、意見が欲しい。

### 3.2 テクニカルスコア (`technicals.py`)

現在の実装: トレンド(MA25乖離率)40% + RSI 25% + 出来高20% +
ゴールデン/デッドクロス15% の加重合成。

**疑わしいと思っている点**:
- 重み(40/25/20/15)も仮置き。テクニカル指標同士の相関が高い
  (トレンドとクロスは似た情報を見ている)ため、実質的な情報量以上に
  「トレンド系」が過大評価されている可能性がある
- RSIの解釈(70超は減点、30未満は加点)はいわゆる逆張り的発想。
  強いトレンド相場ではこの解釈自体が逆効果になりうる
- ボラティリティやATRのような「リスクの大きさ」を一切見ていない。
  スコアが高くても値動きが荒い銘柄をリスク調整せずに同列に扱っている

**Fableへの問い**: 指標間の相関を考慮しないまま単純加重するのは
一般的にどの程度問題か。トレンドフォロー的な解釈と逆張り的な解釈
(RSI)を同じスコアに混在させているのは矛盾していないか。
ボラティリティ/リスクの扱いをスコアに組み込むべきか、
それとも別軸として並記すべきか。

### 3.3 ニューススコア (`news_client.py`)

キーワードマッチによる超簡易センチメント判定(ポジ/ネガ語リストとの
部分一致)。

**疑わしいと思っている点**:
- 皮肉・否定文("下方修正の懸念は払拭された"等)を誤判定する
- 日英で語彙リストの質・量が揃っていない
- 見出しの「量」も情報だが(急に報道が増えた=材料が出た可能性)
  現状は考慮していない

**Fableへの問い**: この程度の簡易実装をどこまで許容すべきか。
「センチメント判定はLLM(Fable自身)に投げて、キーワード判定は
前処理のフィルタ程度に格下げする」という設計変更は妥当か
(コストとレイテンシとのトレードオフも含めて)。

### 3.4 総合スコアの合成 (`scorer.py`)

3軸を単純な加重線形和で合成。

**疑わしいと思っている点**:
- 「ファンダは良いがテクニカルは最悪」のような銘柄と
  「全部中程度」の銘柄が同じ総合点になり得る。線形和だと
  極端な弱点が薄まって隠れる
- 3軸間の重みは利用者が自由に設定できるが、デフォルト値(均等)の
  妥当性を検証していない

**Fableへの問い**: 弱点を隠さない合成方法(例: 最低スコアに
ペナルティを掛ける、下限を設ける等)に変更すべきか。
それとも「総合点」自体を廃止して3軸を独立指標として
併記する設計の方が誠実か。

### 3.5 セクター強度の判定 (`sector_rank.py`)

現在の実装: セクターETF(XLK等)の20営業日リターンからSPYの20営業日
リターンを引いた「相対強度」でランキング。上位N セクターの代表銘柄
(`US_SECTOR_CONSTITUENTS`、各5銘柄程度の仮リスト)だけを個別スクリーニング
の対象にする。

**疑わしいと思っている点**:
- 20営業日という期間は仮置き。短すぎるとノイズを拾い、長すぎると
  もう資金が入り切った後の「出遅れ」を掴む可能性がある
- ETFの相対強度は「時価総額加重平均」なので、大型株1〜2銘柄の動きに
  引っ張られやすい。セクター内の値上がり銘柄の「広がり(breadth)」を
  見ていないため、実際には数銘柄しか上がっていなくてもセクター全体が
  強いと誤判定しうる
- 代表銘柄リストが時価総額上位の有名企業中心の固定リストなので、
  そのセクター内で今まさに動いている中小型の銘柄を拾えない
  構造になっている
- 「セクターが強い→その代表銘柄も強いはず」という前提自体、
  必ずしも成り立たない(セクターは強いのに出遅れている個別銘柄を
  拾いたい、という逆のニーズもある)
- 半導体(SMH)は独立セクターETFではなく情報技術(XLK)の一部という
  分類上のねじれがあり、二重カウントの扱いが未整理

**Fableへの問い**: 相対強度の測定期間(20日)をどう検証・調整すべきか。
ETFベースの相対強度に加えて、セクター内の騰落銘柄比率(breadth)のような
「広がり」の指標を組み込むべきか。代表銘柄を固定リストではなく
動的に(例: そのセクターETFの上位構成銘柄をAPIから取得する等)
決める設計に変えるべきか。

## 4. 今後の拡張方針への疑問

### 4.1 複数証券会社対応(執行系)

想定: kabu STATION API / 立花証券API / moomoo API(再開後)などを
選定結果の「実行アダプタ」として後付けする。

**未決定事項**:
- 発注機能を追加する場合、「選定はしても発注しない」という
  現在の安全設計をどう維持するか。単に別リポジトリ/別プロセスに
  完全分離すべきか、それとも同一プロジェクト内で
  「確認ステップを必須にする」形で共存させるべきか
- 証券会社ごとに扱える商品・注文種別が異なる(信用取引の可否、
  日本株オプションの有無等)。抽象化レイヤーをどこまで頑張るべきか、
  それとも早期に諦めて証券会社別の実装を許容すべきか

### 4.2 リアルタイム化

想定: 現在は日次バッチだが、将来的にはザラ場中のスコア更新も
検討したい。

**未決定事項**:
- リアルタイム化するとJ-Quants/yfinanceの無料枠のレート制限に
  すぐ当たる。有料データソースへの切替判断基準をどこに置くか
- スコアリング自体は重い処理ではないが、Fable 5への問い合わせは
  レイテンシもコストも高い。リアルタイム運用時にFableへの問い合わせ
  頻度をどう制御すべきか(差分が一定以上動いた時だけ問い合わせる、等)

### 4.3 バックテスト機能

**未決定事項**:
- 現状のスコアリング係数(3.1, 3.2の仮置き係数)を過去データで
  検証する仕組みが全くない。これを先に作るべきではないか、
  それとも実運用しながら経験的に調整する方針で良いか

**Fableへの問い**: 4.1〜4.3の中で、次に着手すべき優先順位を
つけるとしたらどうなるか。また、それぞれの「未決定事項」に対する
一次見解が欲しい。

## 5. Fableへの依頼のまとめ

1. 第2章のアーキテクチャ全体について、過剰設計/過小設計の指摘
2. 第3章の各スコアリングロジック(3.1〜3.4)およびセクター強度判定(3.5)
   について、具体的な改善案(可能なら数式・擬似コードレベルで)
3. 第4章の拡張方針について、優先順位と各未決定事項への見解
4. 上記以外で、このドキュメントに書かれていないが設計上
   見過ごされていそうなリスクや論点があれば指摘してほしい

出力は「指摘事項リスト」+「対応する改善案(採用可否は利用者が判断する
ための材料として)」の形式を希望します。
