# stock-selector 設計書

第三者レビュー(Codex等)向けの設計資料。実装の経緯ではなく、要件・設計・判断とその理由をまとめる。
実装履歴の詳細は `notes/CHAT_HANDOFF.md`、運用手順は `README.md` を参照。

- リポジトリ: https://github.com/baw0528-crypto/stock-selector (public)
- モバイル閲覧: https://baw0528-crypto.github.io/stock-selector/ (パスワードゲート)

## 1. 目的・スコープ

米国株を対象に、ファンダメンタルズ・テクニカル・ニュース・セクターローテーションを
組み合わせて銘柄をスコアリングし、投資判断の**材料**となるレポートを生成する。

**絶対条件(変更不可):**
- 売買の発注機能は実装しない。選定・レポート・検証まで
- 出力は投資助言ではなく判断材料。レポートに免責を明記する

**利用者コンテキスト:** 個人投資家1名。元手が少額で、資産成長を目的とする。
参考としている戦略の型は「中小型株 × 決算等の材料 × 短期〜中期モメンタム × 損切り規律」。
ただし参考記事の掲げる月利20〜25%という目標水準はベンチマークにしない(合意済み)。

## 2. 要件

### 機能要件
1. スクリーニング: 指数ユニバース(S&P500/400/600/1500)から2段階選別
   (テクニカル粗選別 → 上位のみフル評価)。任意ティッカー指定・セクター先行モードあり
2. スコアリング: ファンダ/テクニカル/ニュースを各0-100点化し重み付き線形合成(重みはCLI引数)
3. レポート: Markdown(人間用) + JSONスナップショット(検証用)。Claude Fable 5による
   総合コメント(ANTHROPIC_API_KEY設定時のみ)
4. 検証: (a) フォワードテスト=スコアと事後リターンの順位相関、
   (b) ペーパートレード=仮想売買の勝率・平均損益・プロフィットファクター
5. モバイル閲覧: 静的PWA(一覧・詳細・銘柄ドリルダウン・ダッシュボード)。サーバー/API/DBなし。
   詳細画面は同条件の前回レポート比の変化バッジ(NEW/順位変動)付き。ダッシュボードには
   仮想ポートフォリオ(決算またぎ警告付き)・直近の決算予定・フォワードテスト結果を表示
6. 自動運用: 毎朝9時(JST)にスクリーニング→仮想売買判定→暗号化同期→git push

### 非機能要件
- データソースは無料枠(yfinance / Wikipedia / Google News RSS / J-Quants無料)。
  遅延・欠損があり得る前提で設計する
- 欠損データは「中立」に化けさせない(評価不能として明示・除外)
- 再現性: 実行条件(重み・ユニバース・スコア定義バージョン)をスナップショットに記録
- 秘匿: レポート本文はリポジトリに平文で置かない(公開リポジトリのため)

## 3. アーキテクチャ

```
screen.py                     # CLIエントリ。ユニバース解決→スコア→レポート/スナップショット
track_positions.py            # ペーパートレード(仮想エントリー/利確損切り判定/成績集計)
forward_test.py               # スコアと事後リターンの順位相関を集計
sync_report.py                # output/ を暗号化して docs/reports/ へ同期
daily_run.sh                  # 毎朝の自動実行(launchd: com.stock-selector.dailyrun)
src/data/us_market_client.py  # yfinance + 指数構成銘柄(Wikipedia、30日キャッシュ)
src/data/news_client.py       # Google News RSS + キーワードセンチメント
src/data/sector_data.py       # セクターETF/代表銘柄の静的マッピング
src/data/jquants_client.py    # 日本株(休眠中・未検証)
src/analysis/{fundamentals,technicals,scorer,sector_rank}.py
src/agent/fable_synthesis.py  # Fable 5レポート生成(プロンプトインジェクション対策込み)
docs/                         # PWA本体 + 暗号化レポート = GitHub Pages公開ディレクトリ
tests/                        # pytest 40件(ネットワーク不要)
```

データフロー: yfinance/RSS → スコアリング → `output/report_*.{md,json}` →
(暗号化) → `docs/reports/*.enc` → git push → GitHub Pages → ブラウザ内復号。

## 4. スコアリング仕様(SCORE_VERSION = 3)

スコア定義を変えたら`SCORE_VERSION`を上げ、スナップショットに記録する。
forward_testはバージョン混在を警告する(新旧の相関を混ぜないため)。

### ファンダ(最大6指標の平均、取得できた指標のみ)
| 指標 | スコア化 |
|---|---|
| ROE | 0%→30点、20%→90点の線形 |
| 増収率 | -10%→20点、+20%→90点 |
| PER | 15倍前後中立。**赤字(PER≤0)は15点固定**(欠損扱いにしない) |
| PEG | 利益成長>0のときPERスコアと半々ブレンド(高成長の高PERを救済) |
| 利益率 | 線形 |
| D/E | 50%以下健全〜300%超は低評価 |
| 決算サプライズ | **発表45日以内のみ**加味。+5%→75点、ネガティブは大幅減点 |

### テクニカル(重み固定: trend .30 / 相対強度 .20 / 52週高値 .15 / RSI .15 / 出来高 .10 / イベント .10)
- トレンド: 終値のMA25乖離率
- 対SPY相対強度: 60営業日超過リターン(±25%で飽和)
- 52週高値近接度: 高値-30%で0点〜高値で100点
- RSI: **モメンタム解釈**(高いほど加点、85超のみ反落リスクで頭打ち)。逆張り解釈は廃止
- イベント: ゴールデンクロス±10、ギャップ上昇(+3%超の寄り)×出来高2倍超で+15

### ニュース
キーワードセンチメント(単語境界一致)。shrinkage(仮想中立4本)で少数サンプルの振れを抑制。

## 5. 検証設計

- **テクニカル因子のみのバックテスト**(`backtest_technical.py`): ファンダ・ニュースは
  「その時点の値」を無料APIで取得できず先読みバイアスになるため対象外。テクニカルは
  価格履歴から任意の過去時点で再現計算できるため、週次リバランスで過去2年分を
  シミュレーションできる。`track_positions.py`の`evaluate_exit()`をそのまま再利用し、
  ライブのペーパートレードと同一ロジック(利確/損切り/時間切れ)で比較可能にしている。
  2026-07-14、sp600・過去2年・週次リバランス・上位3銘柄で138トレード:
  勝率47.1%、平均損益+0.67%、PF 1.2

- **フォワードテスト**: スコア3分位ごとのN営業日後リターン、SPY比較、Spearman順位相関。
  相関が0近辺ならスコアの並べ替えに意味がない兆し。
  起点は「シグナル確定後に観測できる最初の終値」(JST朝実行のため実質的に翌米国営業日終値)。
  これは先読み防止のための意図的な仕様(次の終値で買う想定に一致)。
  実行条件(ユニバース・重み・スコアバージョン)が異なるスナップショットは混ぜず、
  条件別に集計する。観測は同一銘柄・重複期間で相関するため、表示される観測件数より
  実効サンプル数は少ない(ユニーク銘柄数を併記)
- **ペーパートレード**: エントリー=スナップショットに記録されたシグナル時点の終値
  (`as_of_close`。実際の約定と約1営業日ズレる近似。旧形式スナップショットは実行時の
  最新終値にフォールバック)。利確+10% / 損切り-7% / 20営業日で時間切れ。
  日次OHLC判定で同日両触れは損切り優先、ギャップ越えは寄り値約定(保守側に倒す)。
  価格は調整済みOHLC(分割は正しく処理されるが、保有中に配当落ち日を挟むと
  数値が僅かに歪む。個別配当は損益に含まれない=価格リターンベース)
- **2本立て並行検証**(戦略ごとに状態分離):
  - default: S&P1500全体、重み等分
  - aggressive: S&P600小型のみ、テクニカル0.60/ファンダ0.25/ニュース0.15
- 勝率が統計的に意味を持つのは20〜30トレード以降。判断はそれまで保留

## 6. セキュリティモデル

- GitHub FreeはprivateリポジトリからPages公開不可 → リポジトリはpublic
- レポート本文(銘柄・スコア・コメント)はPBKDF2(SHA-256, 210k)+AES-256-GCMで
  暗号化した`.enc`のみをコミット。復号はブラウザ内Web Crypto APIのみ
- 閲覧パスワードは利用者判断で低強度(利便性優先)。総当たりに弱いことは説明・合意済み。
  守っている資産は「個人の検証データの秘匿」であり、資金・認証情報ではない
- `.env`(パスフレーズ・APIキー)はgitignore。LLMへの外部由来テキスト
  (ニュース見出し・銘柄名・生データ)はタグ囲い+エスケープでプロンプトインジェクション対策済み
- PWAの「この端末を記憶する」はパスフレーズをlocalStorageに平文保存する。
  端末内漏洩・XSS時漏洩のリスクは利便性とのトレードオフとして許容(利用者合意済み)

## 7. 主要な判断記録(ADR相当)

| # | 判断 | 理由 |
|---|---|---|
| 1 | 発注機能は作らない | 誤発注・責任リスク。検証と判断材料に徹する |
| 2 | 欠損≠中立 | データ無し銘柄が中立50点で中位に紛れるのを防ぐ。評価不能として除外表示 |
| 3 | RSIをモメンタム解釈に統一 | trend/出来高/クロスと方向を揃える。逆張り解釈は矛盾していた |
| 4 | 重み・合成方式は当面固定 | フォワードテストの証拠が貯まる前の調整は過学習。因子追加はscore_versionで区別 |
| 5 | 粗選別はテクニカル基準 | 1500銘柄のファンダ取得はAPI的に非現実的。モメンタムバイアスは明記して許容 |
| 6 | リポジトリpublic化 | Pages公開の制約。レポートは暗号化で秘匿し、コードのみ公開 |
| 7 | 決算サプライズは45日以内のみ | 「材料直後」のカタリスト意味論。古いサプライズは状態であって材料ではない |
| 8 | ペーパートレードの約定仮定は保守側 | 同日TP/SL両触れ→SL優先、ギャップ→寄り値。勝率の過大評価を防ぐ |
| 9 | 2本立て並行検証 | アグレッシブ化の要望に対し、既存検証データを汚さず比較データで判断する |
| 10 | 日本株コードは休眠のまま保持 | プロジェクト目的に日本株を含むため退役しない。再開時はJ-Quants検証から |

## 8. 既知の弱点・レビューで意見が欲しい点

1. ニュースセンチメントがキーワードベースで素朴(誤分類が多いはず)。かつ総合点の
   最大1/3を動かせる因子なので影響は軽くない。LLM分類はコスト増とのトレードオフ
2. yfinance非公式APIへの依存(仕様変更・レート制限・データ欠損リスク)
3. 粗選別のモメンタムバイアスがファンダ優良株を第1段階で落とす可能性
4. スコアの線形合成という設計自体の妥当性(因子間の相互作用を表現できない)
5. **ファンダ・ニュースの欠損銘柄が中立50点として総合点に入る**(価格欠損のみ除外)。
   設計方針「欠損≠中立」の部分未実装。修正はスコア定義変更(v4)になるため、
   検証データが貯まってから対応する(意図的な保留)
6. 決算サプライズのスコア化係数(×5)は仮置き。実証的裏付けなし
7. 日本株(J-Quants)経路は全体が未検証

## 8.5 勝率改善の検討(2026-07-14、結論: 現時点でルール変更なし)

「勝率をもっと上げられないか」の検討をCodexと2往復で実施。`backtest_technical.py`に
選別力の検証機能(day_top_score/day_pick_spread/day_cutoff_gap/entry_atr_pctの
3分位分析、ATR%ボラティリティフィルタ)を追加し、sp600・過去2年で検証した。

- **合意した前提**: TP/SL/max_holdをこのバックテストデータに直接チューニングするのは
  オーバーフィッティング。改善はエントリー条件(質)側で検証すべき
- **ATRフィルタ**: 方向としては改善(勝率47.1%→48.6%、PF 1.2→1.29)だが、閾値が
  このデータの中央値由来で効果も小さく、採用の根拠として弱い
- **スコア上位1/3(最も過熱した日)が明確に悪い**(勝率40%・平均-0.92%)という
  気になるパターンを発見。「過熱モメンタムの反落リスク」として理屈は通るが、
  リバランス日は46回しかなく(同日3トレードは同じ値を共有するため実質サンプルは
  138ではなく46)、統計的に弱い
- **結論(Claude・Codex一致)**: 今回の検証では実運用ルールを変更しない。
  次にやるなら (1) リバランス日単位での再集計、(2) 閾値の感度分析(複数点で安定するか)、
  (3) ATR除外銘柄と代替採用銘柄のペア比較、(4) より長い期間での検証、の順

## 8.6 参考記事に寄せた設計の検証(2026-07-16)

「元記事(souzai.net)の設計に寄せる」検証を実施。`backtest_technical.py`に
カタリストゲート(`--require-catalyst`: ギャップ上昇+出来高急増が確認できた
銘柄のみ候補にする)とトレーリング出口(`--exit-style trailing`:
`track_positions.evaluate_exit_trailing()`、含み益+1%でロック後、高値から-5%で
手仕舞い、固定TPなし)を追加。日本株(日経225、yfinance経由、`.T`サフィックスで
価格取得のみなのでJ-Quants未検証問題を回避可能)も追加してUS/JP双方で比較した。

| 条件 | US(sp600) | JP(nikkei225) |
|---|---|---|
| ベースライン(固定TP+10%/SL-7%、無条件) | 47.1% / PF1.20 / 138件 | **57.4% / PF1.83 / 129件** |
| トレーリングのみ(カタリストゲート無し) | 44.9% / PF1.03 / 138件 | (未実施) |
| カタリスト+トレーリング(記事寄せ) | 46.4% / PF1.46 / 138件 | 52.0% / PF1.25 / 125件 |

**結論: 「記事寄せ」の変更は市場によって効果が逆**。USではカタリストゲート+
トレーリングでPFが改善(1.20→1.46)したが、JPでは逆に悪化(1.83→1.25)。
JPは大型株中心(日経225)でボラティリティが低く、固定+10%利確の方がトレンドを
素直に捉えられた可能性がある。**「記事の設計を機械的に移植すれば良くなる」は
誤りで、市場ごとに再検証が必要**という教訓。カタリストゲートは母集団が大きい
(sp600=602銘柄)と毎回条件を満たす銘柄が見つかってしまい、記事が意図する
「頻度が自然に絞られる」効果は出なかった(トレード数は変わらず、質だけ変化)。

現時点でdaily_run.shへの反映はしない(JPは実運用対象外のままだが検証データとして
記録)。テスト14件追加(evaluate_exit_trailing関連、計55件全パス)。

## 8.7 記事の資金管理規律の追加検証、およびJPベースラインの頑健性確認(2026-07-16)

8.6で「元記事の設計をそのまま移植しても市場依存」と分かった後、ユーザーから
「レバレッジ以外はちゃんと寄せたのか」という指摘があり、実際には**寄せきれて
いなかった点(3銘柄までの資金管理規律)**を追加実装。`run_backtest()`に
`max_concurrent`を追加(同時保有N銘柄まで、空きが出るまで新規建てしない。
`ナンピン厳禁`も既存の「保有中銘柄は候補から除外」ロジックで元々満たしていた)。

| 条件(JP) | トレード数/月換算 | 勝率 | PF |
|---|---|---|---|
| ベースライン(固定、無条件、2年) | 129件 / 月約12回 | 57.4% | 1.83 |
| **ベースライン(固定、無条件、4年に延長)** | 423件 / 月約9回 | **56.0%** | **1.75** |
| カタリスト+トレーリング(3枠上限なし、2年) | 125件 / 月約12回 | 52.0% | 1.25 |
| カタリスト+トレーリング+**3枠上限**(2年) | 84件 / **月約8.4回**(記事とほぼ一致) | 51.2% | 1.07 |

3枠上限を足すと頻度は記事の「月5〜10回」にかなり近づいたが、**成績はむしろ悪化**
(PF1.25→1.07)。一方、**ベースライン(無条件・固定ルール)を4年に延ばしても
数字はほぼ維持**(57.4%→56.0%、PF1.83→1.75、トレード数3.3倍)されており、
これは2年間のたまたま良かった結果ではなく、比較的頑健な傾向と見てよい。

**結論**: 元記事のスタイル(カタリスト限定・トレーリング・3枠規律)を精緻に
再現するほど、日本株では成績が悪化する。逆に一番シンプルな「無条件・機械的・
固定TP/SL」構成が、今回試した中で最も良い、かつ頑健な結果だった。
「記事の規律を細かく真似ること」自体に価値があるわけではなく、
市場の値動きの性質(日本の大型株はボラティリティが低く、伸ばすより
早めに確定させる方が合う)に、たまたま今のシンプルな設計が合っていた
可能性が高い。テスト6件追加(max_concurrent関連、計56件全パス)。

## 8.8 日本の小型株ユニバースでの検証(2026-07-16)

「大型株(日経225)以外だと?」という質問に対応し、`get_tse_growth250_tickers()`
(東証グロース市場250指数、273銘柄、米国sp600相当の小型成長株ユニバース)を追加。
元記事は本来この手の小型株を対象にしているため、より直接的な比較になる。

| ユニバース | ルール | 勝率 | PF |
|---|---|---|---|
| 日経225(大型、2年) | ベースライン | 57.4% | 1.83 |
| 日経225(大型、4年) | ベースライン | 56.0% | 1.75 |
| 日経225(大型) | カタリスト+トレーリング | 52.0% | 1.25 |
| 日経225(大型) | カタリスト+トレーリング+3枠規律 | 51.2% | 1.07 |
| **東証グロース250(小型)** | **ベースライン** | **41.1%** | **1.07** |
| **東証グロース250(小型)** | **カタリスト+トレーリング** | **37.2%** | **1.02** |

**予想に反する結果**: 米国では小型株(sp600)でカタリスト+トレーリングが効いたため
「日本の小型株でも同様では」と仮説を立てたが、逆だった。日本は**大型株の方が
小型株より一貫して良い成績**(41.1%より57.4%の方が高い)で、かつ**カタリスト+
トレーリングは大型・小型どちらでも悪化方向**(日経225: 57.4%→52.0%、
グロース250: 41.1%→37.2%)に働いた。米国の「小型株だからこそ記事の設計が効く」
という説明は誤りで、単に米国市場と日本市場の違い(あるいは今回のグロース市場
250銘柄という小さめの母集団・データ品質)による可能性が高い。
